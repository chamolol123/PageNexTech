De dónde nace

El proyecto surge como tesis de estudiantes con el objetivo de construir una solución IoT modular para monitoreo y control en tiempo real. Se prioriza la prueba rápida en laboratorio (Raspberry Pi) y la portabilidad hacia hardware embebido de bajo consumo para despliegues en campo.

¿Por qué microservicios?

  • Portabilidad hacia ESP32: separar funciones en servicios pequeños permite crear builds ligeras (solo lo necesario) para ejecutarse en la ESP32 o en el borde.
  • Escalabilidad y resiliencia: cada servicio se despliega y escala de forma independiente, evitando cuellos de botella del monolito.
  • Mantenibilidad: dominios bien definidos (ingesta, reglas, notificaciones, APIs) permiten ciclos de vida y stacks diferentes sin interferir.

Tecnologías utilizadas

  • Hardware: Raspberry Pi (gateway), sensores/actuadores; destino ligero: ESP32.
  • Mensajería: MQTT como broker; tópicos por dispositivo/servicio.
  • Back-end (servicios): Java/Spring Boot y/o Python/FastAPI en contenedores.
  • Persistencia: PostgreSQL en gateway; SQLite/archivo en despliegues edge.
  • Orquestación: Docker Compose (dev) con miras a K3s/Kubernetes para escalar.
  • Observabilidad: logs estructurados, métricas básicas y tablero de eventos.
  • Front: app web para estado de dispositivos, reglas y control.

Arquitectura (vista rápida)

  • Device Agent: cliente MQTT en el borde (Raspberry/ESP32) para publicar telemetría y recibir comandos.
  • Ingesta: servicio que normaliza mensajes y valida esquemas.
  • Reglas/Alerts: evaluación de umbrales y disparo de notificaciones.
  • API Gateway: expone endpoints para UI e integraciones.
  • UI Web: tablero de dispositivos, eventos y control básico.
